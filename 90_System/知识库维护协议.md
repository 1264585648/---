# 知识库维护协议

## 目标

把视频制作相关的原始材料、会话历史、项目产物和复盘，持续编译成一个可读、可链接、可追溯的 Markdown wiki。这个知识库不是单纯资料仓库，而是给 AI 和人共同使用的工作记忆。

## 三层结构

### 1. Sources

位置：`10_Sources`

用途：保存原始材料的索引、出处、可信度、导入状态。

规则：
- 原始材料尽量不改写。
- 每条来源必须登记到 `10_Sources/来源索引.md`。
- 来源可以是本地文档、网页、论文、工具文档、会话导出、项目复盘。
- 如果内容存在编码问题、过时风险或广告倾向，要在备注里标出。

### 2. Wiki

位置：`20_Wiki`

用途：保存由 AI 编译出来的稳定知识页。

规则：
- Wiki 页应当是可读的结论层，不是原文摘抄。
- 每个关键结论尽量链接到来源编号。
- 同一个主题只维护一个主页面，避免碎片化重复。
- 新材料进入时，优先更新已有页面，而不是新建相近页面。

### 3. Memory

位置：`40_StyleMemory`

用途：沉淀用户的视频风格、偏好、反复出现的判断、工具习惯和制作约束。

规则：
- 风格画像必须区分“观察到的事实”和“推断”。
- 单次会话不能轻易改写长期画像，至少需要多次证据或明确确认。
- 记录负偏好：不喜欢什么、避免什么，和正偏好同样重要。
- 项目复盘是最重要的风格证据。

## 更新节奏

每次有新材料时：
1. 放入 `00_Inbox` 或保留在根目录。
2. 在 `10_Sources/来源索引.md` 登记。
3. 判断是否更新 `20_Wiki`。
4. 如果涉及个人习惯，更新 `40_StyleMemory/制作习惯日志.md`。
5. 如果形成稳定倾向，再更新 `40_StyleMemory/风格画像.md`。

每个视频项目结束时：
1. 完成项目卡。
2. 写一次复盘。
3. 抽取可复用流程、模板、踩坑。
4. 更新风格画像和制作习惯日志。

## 质量标准

好的 wiki 页面应该满足：
- 标题清楚，能被搜索找到。
- 结论先行，细节后置。
- 有来源链接或项目证据。
- 保留适用边界，不把一次经验写成永恒规律。
- 能直接指导下一次视频制作。

## 编码约定

- Markdown 使用 UTF-8。
- 文件名优先中文，便于检索和浏览。
- 代码、命令、工具名保持英文原名。
- 若旧文件显示乱码，先确认编码，不要直接基于乱码内容总结。
