# 从 Markdown 内容生成完整科普视频的工程路径

## 目标

把 `CLI科普视频制作工具推荐.md` 这类长篇研究内容，转成一条可重复执行的视频生产流水线：

`Markdown -> 内容拆解 -> 解说稿 -> 分镜 -> 资产生成 -> 片段渲染 -> 配音字幕 -> 合成后期 -> 质检发布`

这条路径优先采用成熟开源项目，保留人工审核节点，适合生成 3-15 分钟横屏科普视频，也可以裁切为短视频版本。

## 效果最佳主路径

如果目标不是最快落地，而是当前开源方案里尽可能做出最高成片效果，推荐主路径改为：

`Markdown -> 人工审核脚本/分镜 -> Remotion 主视觉系统 -> Manim 科学推演镜头 -> HyperFrames 快速 HTML 特效镜头 -> 专业音频/字幕 -> FFmpeg 精修合成`

一句话：**Remotion 做全片主工程，Manim 做科学表达，HyperFrames 做 agent 友好的 HTML 动效补充，FFmpeg 做最终母版。**

不建议把 HyperFrames 作为唯一主渲染器。它很适合 agent 从 HTML 快速生成高质量镜头，但如果追求整片视觉一致性、复杂组件复用、数据驱动、时间线控制和长期维护，Remotion 当前仍然更适合做主工程。

## 工具分工

| 层级 | 推荐工具 | 作用 |
| --- | --- | --- |
| 主视频工程 | Remotion | 全片时间线、React 组件、数据图表、字幕动画、品牌视觉系统、批量输出 |
| 科学可视化 | Manim Community | 数学、物理、公式、图形推演、算法演示 |
| HTML 特效镜头 | HyperFrames | HTML/CSS/GSAP/Three.js 场景、网页化视觉、agent 快速生成镜头 |
| 3D/高级视觉 | Three.js / React Three Fiber | 结构化 3D 场景、技术架构空间化、粒子/光效 |
| 动画编排参考 | Motion Canvas | 复杂指令式动画、音频同步镜头，可作为局部替代 |
| 后处理 | FFmpeg | 转码、拼接、混音、字幕压制、封装输出 |
| Python 剪辑 | MoviePy | 简单拼接、叠加、自动化剪辑脚本；性能敏感时退回 FFmpeg |
| 语音字幕 | MiniMax 配音 + Whisper | 使用 MiniMax 生成旁白，Whisper 转录词级时间戳，驱动动态字幕 |

## 为什么这样组合

`Remotion` 的优势是成片系统性更强。它把每一帧当作 React 状态的函数，适合做统一设计系统、可复用图表组件、字幕组件、章节模板、片头片尾和批量版本。对这份 Markdown 报告来说，全片会包含大量技术栈对比表、流程图、引用卡片、代码演示和术语解释，Remotion 的组件化能把视觉质量拉得更稳。

`Manim` 负责效果上限里的“科学可信度”。凡是公式、几何、坐标、算法过程、物理仿真，都交给 Manim。这样画面不是普通 PPT 动画，而是接近 3Blue1Brown 风格的严谨演示。

`HyperFrames` 作为补充，而不是主干。它的 HTML 原生和 agent 友好非常适合快速生成网页化动效、GSAP 时间线、shader transition 和 Three.js 小镜头。但整片主工程仍放在 Remotion，避免不同 HTML 镜头之间风格漂移。

`FFmpeg` 必须作为最终合成核心。无论上游片段来自 HyperFrames、Manim、Remotion 还是素材库，最终都统一进入 FFmpeg 做编码、混音和字幕。

## 效果最佳工作流

### 1. 命题驱动，不一次性做完整视频

每条视频先从一个明确命题开始，而不是直接把整份 Markdown 全量转成成片。

命题格式：

```text
这条视频要回答的问题：
核心观点：
目标观众：
预期时长：
输出形式：
```

示例：

```text
这条视频要回答的问题：为什么“视频即代码”会成为科普视频的新生产方式？
核心观点：科普视频的难点不是剪辑，而是把严谨知识转成可复用、可验证、可批量生产的视觉系统。
目标观众：想用 AI 和开源工具做知识类视频的创作者、开发者、教育产品团队。
预期时长：8-10 分钟。
输出形式：横屏深度科普视频，后续可切竖屏短视频。
```

### 2. 先做大纲和内容概要，逐页验收

第一阶段只产出内容，不规划动效、不生成配音、不写渲染代码。

验收顺序：

1. 命题确认。
2. 视频大纲确认。
3. 每一页/每一节内容概要确认。
4. 每一页口播草稿确认。
5. 全片逻辑连贯性确认。

每一页的验收格式：

```yaml
page: 03
title: "Python 与 TypeScript 的分工"
purpose: "解释为什么视频生产需要双技术栈"
key_points:
  - "Python 负责科学计算和精确推演"
  - "TypeScript 负责 Web 动效和动态排版"
  - "两者不是替代关系，而是上下游协作"
draft_narration: "..."
review_status: "pending"
```

只有当用户确认当前页 `review_status: approved` 后，才进入下一页。

### 3. 全部内容验收后，再规划动效和配音

在所有页面的大纲、内容概要和口播稿都确认无误后，再进入第二阶段：

- 为每一页选择视觉类型。
- 为每一页指定 Remotion、Manim 或 HyperFrames。
- 规划字幕样式和关键词高亮。
- 规划 MiniMax 配音参数。
- 规划镜头时长、转场和音乐。

这个顺序可以避免过早进入视觉制作，导致后面因为文案改动而反复推翻动效和配音。

### 4. 先做导演稿，不直接自动生成视频

当前 Markdown 是研究报告，不能直接转视频。先做 4 份中间稿：

- `script.md`：完整口播稿，控制节奏和叙事。
- `storyboard.yaml`：镜头级分镜，规定每个镜头的视觉目标。
- `visual_style.md`：色彩、字体、图形语言、字幕规则、转场规则。
- `asset_plan.md`：哪些画面用 Remotion，哪些用 Manim，哪些用 HyperFrames。

效果优先时，LLM 只做初稿，关键分镜必须人工审一遍。

### 5. Remotion 建全片视觉系统

Remotion 中先做这些基础组件：

- `TitleScene`：章节标题。
- `ConceptCard`：术语解释。
- `CompareMatrix`：Python vs TypeScript、Remotion vs HyperFrames 等对比表。
- `PipelineDiagram`：Markdown 到视频的流程管线。
- `CodeTerminal`：CLI 命令和日志演示。
- `CitationStrip`：引用来源轻量展示。
- `KineticCaption`：词级高亮字幕。
- `ChapterTransition`：统一转场。

这一步决定整片质感。不要让每个镜头各自生成样式。

### 6. Manim 做高价值科学镜头

只挑最能提升质感的镜头用 Manim：

- Video-as-Code 时间轴和对象状态变化。
- Manim 的 `Scene.setup -> construct -> tear_down` 生命周期。
- Mobject/坐标系/公式变换。
- 物理仿真或图形推演示例。

Manim 输出透明背景或统一背景的视频片段，再导入 Remotion 时间线或最终 FFmpeg 合成。

### 7. HyperFrames 做网页化高表现镜头

适合 HyperFrames 的镜头：

- 技术生态地图。
- GitHub 项目卡片瀑布流。
- HTML/CSS/GSAP 文字动效。
- Three.js 背景或轻量 3D 镜头。
- 快速生成多个候选视觉方案。

HyperFrames 镜头需要经过视觉筛选，只保留风格一致的片段进入主片。

### 8. MiniMax 配音决定最终节奏

效果优先时，旁白不是最后补，而是在内容验收之后、正式动效之前定版：

1. 先定口播稿。
2. 使用 MiniMax 生成高质量旁白。
3. Whisper 转录词级时间戳。
4. Remotion 根据词级时间戳驱动字幕和关键词动画。
5. 根据音频节奏微调每个镜头，不让画面平均铺开。

MiniMax 配音在项目中的职责：

- 生成正式旁白音频。
- 根据角色、语速、情绪和停顿生成不同版本供选择。
- 在单页内容确认后，可以先生成样音，但不进入最终混音。
- 全片口播确认后，再统一生成最终音频，避免局部返工。

### 9. 后期母版

最终不要只用默认 render。建议统一进入 FFmpeg 母版流程：

- 响度标准化。
- 降噪、压缩、限制器。
- BGM ducking。
- 统一色彩空间和编码参数。
- 导出审阅版、发布版、竖屏切片版。

## 流水线设计

### 1. 内容解析

输入：

- `CLI科普视频制作工具推荐.md`

输出：

- `brief.md`：视频定位、受众、时长、画幅、风格
- `outline.json`：章节结构
- `script.md`：口播稿
- `storyboard.yaml`：镜头、画面、旁白、资产、时长

处理规则：

- 一级章节转为视频大段落。
- 二级章节转为镜头组。
- 表格转为动态对比图。
- 技术栈对比内容转为矩阵/雷达图/流程图。
- 引用和链接转为片尾参考资料，不直接塞进口播。

### 2. 脚本压缩

原 Markdown 是研究报告，不适合直接朗读。需要生成 3 种脚本：

| 脚本 | 用途 | 建议长度 |
| --- | --- | --- |
| `script-long.md` | 8-15 分钟横屏深度视频 | 1800-3200 中文字 |
| `script-short.md` | 60-180 秒短视频 | 300-700 中文字 |
| `script-chapters.md` | 每章独立视频 | 每章 400-900 中文字 |

当前这份内容建议先做 8-10 分钟横屏版本，结构如下：

1. 开场：为什么科普视频正在变成软件工程
2. 双技术栈：Python 科学计算 vs TypeScript Web 动效
3. Manim：严谨数学和物理可视化
4. HyperFrames/Remotion：Web 原生视频生成
5. FFmpeg/MoviePy/Editly：后处理和自动化合成
6. MiniMax/Whisper：配音、字幕、动态排版
7. 推荐工程架构和选型结论

### 3. 分镜生成

每个镜头使用统一 schema：

```yaml
- id: s03_02
  title: "Manim 的 Scene 生命周期"
  duration: 12
  narration: "Manim 把每个动画组织成 Scene..."
  visual_type: "manim"
  renderer: "manim"
  assets:
    formula: "Scene.setup -> construct -> tear_down"
  notes: "展示三个阶段沿时间轴依次点亮"
```

推荐 `visual_type`：

- `talking_title`：标题和关键词动效，用 HyperFrames。
- `comparison_table`：技术栈对比，用 HyperFrames 或 Remotion。
- `math_animation`：公式推导，用 Manim。
- `architecture_diagram`：流程架构，用 HyperFrames + SVG/HTML。
- `code_demo`：CLI 命令演示，用 HyperFrames。
- `montage`：素材混剪，用 FFmpeg/Editly。
- `caption_scene`：动态字幕镜头，用 Remotion/HyperFrames。

### 4. 资产生成

目录建议：

```text
video-project/
  brief.md
  STORYBOARD.md
  DESIGN.md
  storyboard.yaml
  scripts/
    script-long.md
    script-short.md
  assets/
    audio/
    captions/
    images/
    manim/
    hyperframes/
    remotion/
    broll/
  renders/
    scenes/
    final/
  tools/
    parse_markdown.py
    build_storyboard.py
    render_all.ps1
```

资产策略：

- 在内容逐页验收完成前，不生成最终旁白。
- 逐页口播稿确认后，可用 MiniMax 生成样音；全片文案锁定后，再统一生成最终旁白。
- 字幕使用 Whisper 生成词级时间戳。
- 关键图表尽量由代码生成，避免手工截图。
- 开源项目 Logo、GitHub 页面截图要检查授权和使用场景。
- 科普示意图尽量用 HTML/SVG/Manim 绘制，减少版权风险。

### 5. 渲染策略

推荐渲染优先级：

1. `HyperFrames`：主视觉、标题动效、流程图、对比图、HTML 场景。
2. `Manim`：公式、几何、物理、算法推演。
3. `Remotion`：需要 React 组件、批量模板、复杂数据绑定时使用。
4. `FFmpeg`：统一拼接、混音、字幕、转码。

单个镜头只选择一个主渲染器，避免维护成本失控。

### 6. 合成后期

最终合成顺序：

1. 标准化所有片段为同一规格：`1920x1080 / 30fps / h264 / yuv420p`
2. 合并场景片段
3. 混入旁白
4. 混入背景音乐和音效
5. 生成软字幕 `.srt/.vtt`
6. 可选压制硬字幕
7. 导出发布版和审阅版

建议产物：

```text
renders/final/video_1080p_review.mp4
renders/final/video_1080p_publish.mp4
renders/final/video_vertical_cut.mp4
renders/final/subtitles.srt
renders/final/transcript.md
```

## 最终选型结论

效果最好的方案是：

```text
Remotion 主工程
+ Manim 精确科学动画
+ HyperFrames 高表现 HTML/Three.js 镜头
+ Whisper 词级字幕
+ MiniMax 高质量配音/真人配音
+ FFmpeg 专业母版合成
```

不把 HyperFrames 放在主工程位置，原因是全片级别的视觉一致性、组件复用、数据驱动和时间线治理，Remotion 更成熟。

不把 Manim 放在主工程位置，原因是它适合科学推演，不适合承担全片排版、字幕、图表、转场和素材混剪。

不把 MoviePy/Editly 放在主工程位置，原因是它们更像后处理和自动剪辑工具，视觉表现上限不如 Remotion/Manim/HyperFrames。

VibeFrame 可以作为后续上层编排工具，但在追求效果的第一阶段，不建议让它完全自动决定分镜和视觉。更好的做法是先人工定导演稿，再让工具执行。

## 分阶段验收版本

本项目不采用“一次性全部完成”的模式。正确流程是先把内容做准，再做视觉和声音。

### 阶段 A：命题与大纲

产物：

- `topic.md`：命题、核心观点、目标观众、预期时长。
- `outline.md`：视频大纲。
- `content_summary.md`：每页内容概要。

验收标准：

- 命题明确，不是泛泛介绍工具。
- 大纲有清晰递进关系。
- 每一页只解决一个表达任务。
- 内容概要经用户确认后再进入下一页。

### 阶段 B：逐页口播稿

产物：

- `pages/page_01.md`
- `pages/page_02.md`
- `pages/page_03.md`

每页包含：

- 标题。
- 本页要讲清楚的一句话。
- 关键论点。
- 口播草稿。
- 用户修改意见。
- 验收状态。

验收标准：

- 用户逐页确认。
- 未确认页面不进入动效规划。
- 已确认页面可以锁定为 `approved`。
- 若后续修改已确认页面，需要重新标记为 `needs_review`。

### 阶段 C：动效与配音规划

触发条件：所有页面口播稿均已确认。

产物：

- `storyboard.yaml`：镜头级分镜。
- `visual_plan.md`：每页视觉方案。
- `voice_plan.md`：MiniMax 配音方案。
- `caption_plan.md`：字幕和关键词高亮方案。

验收标准：

- 每页都有明确渲染器：Remotion、Manim 或 HyperFrames。
- 每页都有明确旁白、时长、视觉目标和转场目标。
- MiniMax 配音参数在全片口播确认后统一规划。

### 阶段 D：制作与渲染

触发条件：动效和配音规划确认。

产物：

- Remotion 主工程。
- Manim 科学动画片段。
- HyperFrames 高表现镜头。
- MiniMax 最终旁白。
- Whisper 词级字幕。
- FFmpeg 母版输出。

验收标准：

- 先交付审阅版，不直接交付发布版。
- 审阅版确认后再做最终混音、字幕压制和编码。

## 后续自动化升级

第二阶段：

- 增加 `render_all.ps1`，按 storyboard 自动调用 HyperFrames、Manim 和 FFmpeg。
- 增加 `validate_storyboard.py`，检查镜头 id、资产路径、时长和字幕文件是否完整。
- 增加 `review_report.md`，记录画面重叠、字幕溢出、音量过大、黑屏等问题。

第三阶段：

- 接入 VibeFrame 作为上层项目编排。
- 把常用镜头模板沉淀为 HyperFrames 模板库。
- 把 Manim 场景封装为可参数化 Python 模块。
- 生成横屏、竖屏、封面图、短视频切片和社媒文案。

```powershell
python tools/build_outline.py --topic topic.md --out outline.md
python tools/build_page_summaries.py --outline outline.md --out content_summary.md
```

阶段 C 确认后，才初始化和执行渲染相关工程：

```powershell
# Remotion 主工程
npx create-video@latest remotion-main

# HyperFrames 辅助镜头
npx hyperframes init hyperframes-scenes

# Manim 科学镜头
manim -qh manim/scenes/core_scene.py CoreScene

# HyperFrames 镜头渲染
npx hyperframes render --output renders/scenes/web_intro.mp4

# 最终母版合成
ffmpeg -f concat -safe 0 -i renders/scenes/list.txt -c:v libx264 -pix_fmt yuv420p renders/final/video_1080p_publish.mp4
```

MiniMax 配音和 Whisper 字幕在阶段 C 规划确认后接入：

```powershell
# 示例：MiniMax 接入脚本
$env:MINIMAX_API_KEY="your_minimax_api_key"
node tools/minimax_tts.mjs --api-url https://api.minimaxi.com/v1/t2a_v2 --input scripts/final_narration.md --out assets/audio/narration.mp3
python tools/whisper_align.py --audio assets/audio/narration.wav --out assets/captions/words.json
```

## 推荐推进顺序

1. 用户提供一个命题。
2. 先产出大纲。
3. 大纲确认后，产出每一页内容概要。
4. 一页一页验收概要和口播稿。
5. 全部内容验收后，再规划动效。
6. 动效方案确认后，再规划 MiniMax 配音。
7. 最后进入 Remotion/Manim/HyperFrames 制作和 FFmpeg 母版合成。

这条顺序优先保证内容正确和成片质感，不以最快生成完整视频为目标。
